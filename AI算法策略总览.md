# 五子棋 AI 算法策略总览

> 本文档总结本项目的完整 AI 算法策略，涵盖传统算法引擎与 LLM 多智能体系统的协同设计。

---

## 一、总体架构：双层混合决策

```
输入：15×15 棋盘状态
        │
        ▼
┌─ 第一层：传统算法引擎 ─────────────────────────────┐
│  · 棋型检测（pattern.py）                          │
│  · 局面评估（evaluate.py）                         │
│  · Minimax + Alpha-Beta 搜索（search.py）          │
│                                                    │
│  职责：精确计算、零延迟、不依赖外部 API              │
│  · 必胜点 → 直接落子                               │
│  · 必堵点 → 强制封堵                               │
│  · 双重威胁 → 直接进攻                             │
│  · LLM 失败 → 搜索兜底                             │
└────────────────────────────────────────────────────┘
        │ 无强制落子
        ▼
┌─ 第二层：LLM 多智能体系统 ─────────────────────────┐
│  · 战术官 + 防守官（并行分析）                      │
│  · 反对官（压力测试，仅复杂局）                     │
│  · 总策划官（汇总裁决 + 技能调用）                  │
│                                                    │
│  职责：战略推理、局面理解、技能运用                  │
│  · 进攻路线规划                                    │
│  · 防守优先级判断                                  │
│  · 技能时机决策                                    │
│  · 开局定式匹配（RAG 棋谱检索）                    │
└────────────────────────────────────────────────────┘
        │
        ▼
输出：(x, y) 落子坐标 + 可选技能激活
```

## 二、第一层：传统算法引擎

### 2.1 棋型检测（pattern.py）

**核心思路**：对某个空位，检查四个方向（水平、垂直、两条对角线）上的连续同色子数量及两端状态，精确判定棋型。

```
方向定义：
  (1, 0)  水平
  (0, 1)  垂直
  (1, 1)  主对角线
  (1, -1) 副对角线

对每个方向：
  正向计数 → 遇边界/对方子/空位/封锁位停止
  反向计数 → 同上
  总长度 = 正向 + 反向 + 1（含自身）
  两端状态 → 各端是"空位"还是"被堵"
```

**棋型映射表**：

| 连续子数 | 两端空位数 | 棋型 | 威胁等级 |
|----------|-----------|------|----------|
| ≥5 | ≥1 | 成五（win） | 立即获胜 |
| 4 | 2 | 活四（live4） | 必胜（对手无法阻挡） |
| 4 | 1 | 冲四（rush4） | 严重威胁（必须阻挡） |
| 3 | 2 | 活三（live3） | 中度威胁 |
| 3 | 1 | 冲三/眠三（rush3） | 轻度威胁 |
| 2 | 2 | 活二（live2） | 潜在威胁 |


**关键函数**：

| 函数 | 功能 |
|------|------|
| `extract_patterns(grid, x, y, player)` | 检测 (x,y) 落子后产生的所有棋型 |
| `find_immediate_win(grid, player)` | 找到当前玩家一步获胜的落点 |
| `find_must_block(grid, opponent)` | 找到对手必胜点（我方必须封堵） |
| `find_double_threat_moves(grid, player)` | 找到形成双重威胁的落点（双活三/活三+冲四/活四） |
| `scan_threats(grid, player)` | 全盘扫描某玩家的所有威胁并分类 |

### 2.2 局面评估（evaluate.py）

**评分公式**：

```
局面分 = AI进攻分 - 对手进攻分 × 0.9
```

**权重表**：

| 棋型 | 权重 | 说明 |
|------|------|------|
| win（成五） | 1,000,000 | 绝对优先 |
| live4（活四） | 50,000 | 必胜，对手无法阻挡 |
| rush4（冲四） | 10,000 | 对手必须阻挡 |
| live3（活三） | 5,000 | 下一手可变活四 |
| rush3（冲三） | 500 | 有限威胁 |
| live2（活二） | 100 | 基础布局 |

**防守系数 0.9**：体现"进攻优于防守"的策略直觉——同等棋型下，AI 优先选择进攻而非被动防守。

### 2.3 Minimax + Alpha-Beta 搜索（search.py）

**搜索策略**：

```
入口：get_best_move(grid, ai_symbol, depth=2, time_limit=0.3s)

候选生成：
  · 仅搜索已有棋子周围 2 格内的空位（大幅剪枝）
  · 候选数：开局 1~3 个 → 中盘 20~80 个 → 残局递减

递归搜索：
  minimax(grid, depth, alpha, beta, maximizing)
    · 终止条件：depth=0 或 检测到赢棋
    · 赢棋加分：WIN_SCORE × (depth + 1)，越早赢分越高
    · 输棋扣分：-WIN_SCORE × (depth + 1)，越早输扣分越多
    · Alpha-Beta 剪枝：β ≤ α 时剪枝

候选排序：
  · 按快速评估降序排列 → 最大化剪枝效率
  · 先检查每个候选是否直接获胜 → 直接返回
```

**复杂度分析**：

| 阶段 | 候选数（半径2） | depth=2 节点数 | 耗时估算 |
|------|----------------|---------------|----------|
| 开局（1-4手） | 3~8 | 9~64 | < 1ms |
| 中盘（10-30手） | 20~50 | 400~2500 | < 50ms |
| 残局（30+手） | 30~80 | 900~6400 | < 200ms |

**超时保护**：每个节点检查累计耗时，超时即返回当前最佳，确保 UI 不卡死。

## 三、第二层：LLM 多智能体系统

### 3.1 四个智能体

| 角色 | 模型 | 职责 | 视角 |
|------|------|------|------|
| 战术官 | qwen-plus | 进攻分析：寻找活二→活三→四三杀的进攻链 | 己方 |
| 防守官 | qwen-plus | 防守分析：识别对手威胁，优先级排序 | 对手 |
| 反对官 | qwen-plus | 压力测试：检查遗漏的威胁、斜线盲区、VCF链 | 批判 |
| 总策划官 | qwen-plus | 综合裁决：汇总所有意见，做出最终落子决定 | 全局 |

### 3.2 决策流程

```
1. GameInfoExtractor 生成 game_report
   ├── 回合信息、棋子坐标
   ├── 15×15 文本棋盘
   ├── 【算法威胁分析】（注入 LLM）
   └── 双方招式说明

2. LangGraph 编排
   ├── phase_check：局势分级
   ├── [tactical || defense]：Send fan-out 并行
   ├── post_analysis（join）：汇总
   │   ├── 共识（同坐标 + 置信度 > 0.8）→ 跳过后续
   │   ├── complex → devil_advocate → chief
   │   └── normal → chief
   └── chief：最终裁决 + 可选技能 tool-calling

3. 校验-重试循环（最多 3 次）
   └── 全部失败 → 算法搜索兜底
```

### 3.3 LLM 上下文设计

**game_report 中的算法威胁分析示例**：

```
【算法威胁分析】
!!! 必须封堵: (7, 3) —— 对手落子即五连！
对手可形成活四的点: (5, 7) (9, 7)
对手可形成冲四的点: (3, 3) (11, 3) (7, 11)
己方可形成活三的点: (8, 6) (6, 8) (10, 4)
```

这确保了 LLM 智能体始终基于精确的算法分析做决策，而非纯靠"目测"，大幅减少漏看致命威胁的错误。

### 3.4 共识短路

当战术官和防守官建议相同落子，且双方置信度均 > 0.8 时，认定该落子为明确最佳选择，跳过反对官和总策划官，直接采用。此优化可将典型回合的 LLM 调用从 4 次降至 2 次。

## 四、辅助系统

### 4.1 RAG 棋谱检索

- **存储**：100+ 专业五子棋开局棋谱 → ChromaDB 向量库
- **检索**：根据当前局面棋子序列，在开局阶段（前 10 步）自动检索 top-3 相似棋谱
- **作用**：为 LLM 提供专业开局参考，弥补纯 LLM 在标准定式上的知识盲区

### 4.2 快速开局通道（turn 1-4）

| 回合 | 策略 | 原因 |
|------|------|------|
| turn 1 | 硬编码天元 (7,7) | 职业规则：先行必占天元 |
| turn 2-4 | 单次 LLM 快速决策 | 开局变化少，无需完整多智能体流程 |

### 4.3 技能 Tool-Calling

总策划官可通过 OpenAI function-calling 协议激活己方主动招式（万宁阵法/血狱影杀阵/四方阵），分析阶段只预览不执行，落子后由 GameController 实际执行。

## 五、决策优先级总览

```
优先级 1：成五获胜
  └→ 算法预检检测到 win ≥ 1 → 直接落子

优先级 2：封堵对手成五
  └→ 算法预检检测到对手 win ≥ 1 → 强制封堵

优先级 3：己方双重威胁
  └→ 算法预检检测到双活三/活三+冲四/活四 → 直接进攻

优先级 4：活四/冲四威胁
  └→ 算法分析注入 game_report → LLM 综合判断

优先级 5：活三/活二布局
  └→ LLM 主导的战略规划

优先级 6：无威胁局面
  └→ LLM 开局定式 + RAG 参考 + 自由发挥
```

## 六、与传统纯算法 AI 的对比

| 维度 | 传统 Minimax AI | 本项目混合 AI |
|------|----------------|--------------|
| 必胜/必堵检测 | ✓ 精确 | ✓ 精确（算法层） |
| 深度搜索 | depth 4-6+ | depth 2（辅助用） |
| 开局知识 | 依赖开局库 | RAG 棋谱 + LLM 推理 |
| 战略理解 | 无 | LLM 可进行语义级局面理解 |
| 技能运用 | 不可用 | LLM 判断技能使用时机 |
| 适应性 | 固定策略 | LLM 可适应非标准局面 |
| 响应速度 | 快（纯计算） | 算法层 < 1ms，LLM 层 2-8s |
| 透明度 | 高（可解释） | LLM 输出推理文本，可追溯 |

**设计哲学**：传统算法处理"精确计算"（什么是威胁，哪里必须落子），LLM 处理"语义理解"（为什么这样走，什么时候用技能）。两者互补而非替代。
